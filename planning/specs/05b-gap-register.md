# Spec 05b ‚Äî Gap Register (evolving)

**Status:** v0.1 ‚Äî July 2026 ‚Äî live worklist. Itemizes every gap/seam the end-to-end traces ([`05a-traces.md`](05a-traces.md)) surface where the upstream specs are silent, contradictory, or under-powered. This is the input to the **resolve stage**: each gap gets a designed solution that is then written back into specs 01‚Äì05, after which the vertical examples are re-confirmed end-to-end.

**Why separate from 05a-traces:** the traces are a *narrative* (what happens, step by step); this is a *ledger* (discrete items with IDs, severity, and status that evolve independently as we design fixes). Kept apart so the trace prose stays readable and each gap is trackable through resolution.

**Lifecycle of a gap:** `open` (surfaced) ‚Üí `designing` (resolve stage, solution being specced here) ‚Üí `resolved` (written into the target spec; cite section). Severity: **blocker** (a vertical example cannot function without it) ¬∑ **major** (works but a real design decision / possible re-architecture) ¬∑ **minor** (local convention or wording).

> ‚öÝ Nothing here is folded into the upstream specs yet (Luis's call ‚Äî gaps may still evolve as more examples are traced). Fold-back happens in the resolve stage.

## 2. Detail ‚Äî the meta-schema gaps (need design in the resolve stage)

These three (G-10, G-11, G-12) are the ones that may require *changing the meta-schema*, not just writing a missing section ‚Äî so they carry the most implementation risk and get first attention in the resolve stage.

### G-10 ¬∑ Arbitrary personal facts
Types have fixed attribute schemas (Spec 01 ¬ß4). Real people-knowledge is open-ended ("allergic to peanuts", "hates cilantro", "middle name is Rose"). Options to design:
- **(a) Generic `contact_fact` child type** ‚Äî `parentType: contact`, attributes `{attribute?, value?, fact}` (structured when extractable, free-text otherwise). Queryable via `read_related`; `recall-contact-fact` reads it. *(Traces adopt this provisionally.)*
- **(b) A `json` fact-bag attribute on `contact`** ‚Äî simplest, but blocks structured querying (Spec 01 ¬ß3 warns against `json`).
- **(c) Per-fact attributes authored on demand** ‚Äî turns every new fact into a (paid) schema edit; wrong for a free-tier capture.
- **Decision driver:** F-08 recall ("What's Mia allergic to?") wants queryability ‚Üí leans (a). Confirm the type and whether it's `append`-only.

### G-11 ¬∑ Qualified relationships
The kernel Relation can't say *how* two contacts relate. Options:
- **(a) `contact_relationship` record type** `{fromContact‚Üícontact, toContact‚Üícontact, relationType: text}` ‚Äî matches how NLU actually extracts (`relationType:"daughter"`), symmetric/queryable, free-text roles. *(Traces adopt this provisionally.)*
- **(b) Add a `role`/qualifier field to the kernel Relation** ‚Äî keeps it in the graph's `relations` array (Spec 01's "independently queryable graph"), but is a **kernel change** (version bump) and still needs a controlled vocabulary or free-text role.
- **(c) A fixed set of named relations on `contact`** (`children`, `parents`, `partners`, ‚Ä¶) ‚Äî coarse, and "cousin/colleague/mentor" don't fit.
- **Decision driver:** keeping the relationship graph first-class (b) vs. modeling it as records (a). (a) is less invasive; (b) is more faithful to the graph vision. Resolve-stage call.

### G-12 ¬∑ Resolve-or-create + name uniqueness
Proposed contract to design and state:
- **NLU owns disambiguation of *existing* entities.** `entityNames.resolve('contact', "Sarah")` ‚Üí if exactly 1, pass `sarahId`; if >1, **clarify before dispatch** ("Which Sarah ‚Äî Mitchell or Chen?", Spec 05 ¬ß9 E2); if 0, pass the *name* for the skill to create.
- **The skill owns create-if-absent** (a write, so it must be in execute): `read_one contact {displayName}` ‚Üí `branch null` ‚Üí `write_record contact`. This means a seed skill takes **both** an optional resolved `‚Ä¶Id` *and* a `‚Ä¶Name` per person, or a single name slot with the skill always doing resolve-or-create. Pick one and make it the idiom.
- **`read_one` on a non-unique field** must be defined: error, or "first by recency," or forbidden (require id). Given NLU pre-disambiguates, the skill's `read_one {displayName}` should only ever see 0-or-1 ‚Äî but that invariant must be stated and enforced.

## 3. Planned ‚Äî Local-model trust evaluation (G-20)

To be built **after 5a is complete** (Luis's direction). Extends the existing rig ([`05a-rig/harness`](05a-rig/harness)). Determines, against a bar set *in advance*, whether the on-device generative model earns a place in the routing path ‚Äî or is cut for deterministic/rule routing + Haiku fallback. This is the rigorous form of D-B and the go/no-go for any on-device generative routing.

**What it tests** (labeled dataset, many paraphrases per case):
- **A. Known-capability routing** *(the real job)* ‚Äî utterance + a realistic *retrieved* candidate set (‚â§5) ‚Üí expected `skillId` + slots. The core metric.
- **B. Slot extraction** ‚Äî exact + normalized precision/recall, including multi-entity (F-07-style).
- **C. Meta-intent detection** ‚Äî novel-need recognition; measured to *confirm it should be retrieval-owned, not model-owned*.
- **D. Out-of-domain boundary** ‚Äî records-vs-OOD, including the adversarial "what did I say about ‚Äπworld-noun‚Ä∫" cases (`G-19`).
- **E. Ambiguous / adversarial** ‚Äî near-miss candidates, coordinations ("salt and pepper"), anaphora ("again", "the usual").

**Models:** Qwen2.5-1.5B, Llama-3.2-3B, Gemma-2-2B, Phi-3.5-mini, + one 7‚Äì8B (ceiling) ¬∑ **Haiku 4.5 as the cloud reference.** **Conditions:** free-form vs grammar/JSON-schema constrained decoding.

**Metrics:** routing accuracy (A) ¬∑ slot P/R (B) ¬∑ format-valid rate ¬∑ latency p50/p95 on target hardware ¬∑ **calibration** (does confidence separate right from wrong? ‚Äî so far *0.0 on correct*, so probably not; if confirmed, escalation must gate on **retrieval** signals, never model confidence).

**Acceptance bar (illustrative ‚Äî pin before running):** routing accuracy ‚â• ~95% among ‚â§5 candidates ¬∑ slot-exact ‚â• ~90% ¬∑ p95 < ~1.5 s. **Decision:** meet the bar ‚Üí the small model routes *known* capabilities only; miss it ‚Üí **cut it** from routing (deterministic/rule where a code path exists, Haiku fallback where none does).

## 4. Resolutions (design decisions ‚Äî resolve stage)

Calls made 2026-07-05 (Luis delegated).
